# fix-0002

## Summary

This branch corrected the mock session model so project and feature screens do not show panes from unrelated projects. The visible bug was that the second mocked project displayed panes from the first project, which made the mobile hierarchy feel untrustworthy even though the app was still mock-backed.

## What Changed

- Added explicit ownership metadata to mocked sessions.
- Added project-scoped and feature-scoped session lookup methods to `TmuxAgentRepository`.
- Updated project detail screens to request only sessions owned by the current project.
- Updated feature detail screens to prefer sessions owned by the current feature.
- Kept a fallback from feature sessions to project sessions so a feature can still use a one-level-up project terminal when no feature-specific session exists.
- Added test coverage for project and feature session scoping.

## Why

The app is meant to navigate from project to feature to terminal context. Showing global tmux sessions in every project breaks that mental model and makes it too easy to send input to the wrong pane. That is especially risky on mobile because the terminal tab is intentionally persistent and reachable from anywhere.

The repository boundary is the right place for this rule. Views should not need to remember how to filter sessions; they should ask for the sessions available to the workspace they are rendering.

## Decisions

- Mock session ownership stays outside the generated OpenAPI-shaped `Session` model for now because the backend contract does not yet expose project or feature ownership on session responses.
- Feature session lookup falls back to project sessions. This keeps the app useful for cases where there is one shared project session and feature-specific session creation has not happened yet.
- The mock data now includes distinct sessions and panes for the iOS project so manual sanity checks can catch cross-project leakage.

## Follow-Up

The backend contract should eventually make session ownership explicit. The iOS repository methods already model the desired caller shape, but the OpenAPI schema will need an ownership field or scoped session endpoints before the real client can enforce this without local inference.
