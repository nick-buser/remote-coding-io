# docs-0003

## Summary

This branch documented the current functional gap between the backend, web client, and iOS client. The gap matters because mobile is intentionally not a web clone, but it still needs parity on the core coding loop: project context, feature context, session/pane selection, output streaming, and reliable input.

## What Changed

- Added `docs/project_plans/mobile_gap_analysis.md`.
- Compared the current OpenAPI backend contract with the web API client and terminal UI.
- Compared the web client against the current mock-backed iOS prototype.
- Called out backend contract gaps that block real mobile persistence for docs, prompts, acceptance criteria, tickets, decisions, session ownership, and transcripts.
- Proposed an implementation order for mobile parity.

## Why

The iOS app already has enough mock UI that it can create a false sense of completeness. Writing down the gaps now keeps the next work grounded: generated API client first, real project/feature data second, terminal transport third, then richer document and ticket surfaces once the backend contract exists.

## Decisions

- The doc treats OpenAPI as the backend source of truth, not the implementation internals.
- Web parity is scoped to behavior, not visual layout. Mobile should borrow the web terminal semantics without inheriting the web desktop UI.
- The session ownership issue from `fix-0002` is listed as a backend contract gap, since mobile can only infer scoped sessions until the API exposes ownership.

## Notes

No code was changed in this branch. The plan is intended as a living project-plan doc and should be updated when backend docs/tickets/session ownership endpoints land.
