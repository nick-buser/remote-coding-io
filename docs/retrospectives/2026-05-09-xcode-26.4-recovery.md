# Retrospective: Xcode 26.4 build recovery

**Date:** 2026-05-09
**Branch:** `main` (working-tree only; nothing committed at the time of writing)
**Outcome:** Build green. Tests blocked by a separate iOS Simulator launch issue.

This is a blameless reflection on the recovery loop — what was broken, why it
got that way, where the iteration was efficient, and where the system can be
hardened so the next time costs less.

---

## What we set out to do

Get the `remote-coding` iOS app building cleanly under Xcode 26.4 / iOS 26.4
SDK on the contributor's local machine, and ideally green on `xcodebuild test`.

## What we actually did

Touched 19 files. Build went from a wall of compiler errors to **BUILD
SUCCEEDED**. Tests can compile but can't actually execute — the simulator is
denying `xctrunner` launches (`SBMainWorkspace` request denied, `RBSRequest
Code=5 Launch failed`), almost certainly fallout from two ENOSPC events that
happened mid-iteration.

The full file-level change list is in the previous turn's summary; this doc
focuses on **themes**, not line items.

## Root-cause patterns

The 19-file diff looks like a swarm of unrelated bugs, but it factors cleanly:

1. **A forced toolchain bump met an unprepared codebase.** Runestone 0.5.2's
   release notes literally say "Fix compilation on Xcode 26.4." The dep bump
   wasn't optional, but it arrived simultaneously with the SDK upgrade, so the
   API migration had to happen *while* the build was rotting.
2. **Generated types changed contract without a caller audit.** Optionality
   flips (`Ticket.estimate`) and protocol additions (`getAgentSession`) are
   exactly the kind of change that should grep for callers before merging.
3. **Stubs and tests are the slowest things to notice rot in.** They don't
   exercise the runtime path, so the only feedback is the next compile —
   which here landed during a crisis instead of in a normal iteration.
4. **Local-environment hygiene is part of the build.** Two ENOSPC stalls,
   then a wedged simulator daemon, made the recovery loop ~3× slower than the
   actual fixes warranted.

---

## What went well

- **Each individual fix was minimal-scope.** No drive-by refactors, no
  speculative abstractions. The diff reads as "the smallest change that makes
  the compiler happy in each spot," which is the right mode for a rotting
  build.
- **No safety bypasses.** No `--no-verify`, no skipping macro validation
  unsafely, no `xcrun simctl erase all` without explicit confirmation.
- **Failure honesty at the end.** When the simulator wedged, the report said
  "build green, tests blocked by simulator infra" instead of overclaiming.
- **User confirmed before destructive cleanup steps** instead of guessing at
  acceptable blast radius.

## What didn't go well

- **ENOSPC was discovered reactively, not preflight.** A `df -h /` at the
  start of `/ios-gates` would have caught it before the build ate the disk.
- **Test launch failures were initially read as test failures.** Forty
  "failed (0.000 seconds)" results are a launch-infra signal, not a test
  signal. Recognizing that earlier would have saved one full test cycle.
- **Themes were discovered late.** Several errors were "the same Runestone
  API rename in a different file." Scanning for the *theme* (which API
  changed?) before fixing line-by-line would have collapsed multiple passes
  into one.
- **Stub conformers exist at all.** `EmptyInboxRepository` only exists for
  previews. A real `MockInboxRepository` (which is also useful for tests)
  would have made protocol drift loud at compile-time across more sites, not
  fewer.

## Action items

Concrete, ordered by ROI:

1. **`/preflight` and `/ios-gates` should fail fast on low disk.** Add a
   `df -h /` check; bail with guidance if free space is below a threshold
   (suggest 10 GiB to leave headroom for a clean build + sim install).
2. **Add a SwiftLint rule (or `git grep` pre-push check) banning curly
   quotes and en-dashes in `.swift` files** outside `Resources/`. The 52-
   quote bug should never have been possible to commit.
3. **Add a SwiftLint rule for `@State var body` / `@StateObject var body`
   /** any property named `body` on a `View` that isn't the actual body.
4. **Delete the preview-only `Empty*Repository` stubs.** Make previews use
   the real `Mock*Repository` instances. Protocol additions then break
   visibly across the app, not just in production conformers.
5. **iOS CI on PR.** Run `xcodebuild build` against a pinned simulator and
   Xcode version on every PR. The user shouldn't be the one who first sees a
   build break against a bumped SDK.
6. **Codegen-change checklist in the OpenAPI workflow.** When `make check-
   generate` produces a diff that flips optionality, adds a protocol method,
   or renames a field, require a `git grep` audit of callers in the same PR.
7. **Treat forced dep bumps as their own ticket.** When an SDK upgrade
   forces a major dep version (Runestone here), the bump and API migration
   land in a dedicated branch with its own gates, *not* mixed into whatever
   work happens to be open.
8. **Auto-recover wedged simulators.** When `xcodebuild test` returns a
   wave of `(0.000 seconds)` results, the gate script should run
   `xcrun simctl shutdown all` and retry once before giving up.
9. **In recovery mode, scan for themes first.** When >5 errors are open,
   pause and bucket them before fixing — multiple call-sites of the same
   API rename should be fixed in one pass.
