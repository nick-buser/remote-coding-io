# docs-0005: Update AGENTS.md and claude.md to reflect the 5-tab v2 shell

Ticket: `.tickets/done/docs-update-agents-shell.md`

## Summary

Rewrote `AGENTS.md` and `ios_apps/claude.md` so the contributor-facing
docs match the v2 design before any screen tickets begin. The previous
prose described a 3-tab shell (`Projects / Terminal / Settings`) with
the terminal as a top-level tab, plus a stale `AppRoute` enum and a
truncated API surface table. Both files now describe the 5-tab shell
(Inbox / Projects / Roadmap / Sessions / You) with the terminal
reframed as a full-screen drill-down, list the full OpenAPI surface
the design relies on (Tickets, Acceptance Criteria, Docs, Decisions,
Activity, Agent Sessions, Review), and link to the v2 design bundle
plus `docs/feature_plans/00-overview.md` as the authoritative entry
points.

This ticket closes Phase 0 of the v2 plan. After this lands, the
Phase 1 navigation tickets (`service-tab-shell` /
`service-app-route-coordinator`) and the Phase 2 repository tickets
can move in parallel without contributors building to a stale shape.

## Changes

- `AGENTS.md` and `claude.md` rewritten in lockstep — they remain
  near-duplicates by design. Both now describe the 5-tab shell, the
  full information hierarchy (Project ▸ Feature ▸ Tickets / Docs /
  Decisions, with Agent Sessions bound to tickets), and the terminal
  as a full-screen drill-down that hides the tab bar while presented.
- `AppRoute` examples updated: `agentSession(sessionID:)`,
  `ticketDetail(publicID:)`, `docDetail(docID:)`, `review(...)`,
  `featureDetail(...)`, `projectDetail(idOrSlug:)`,
  `sessionDetail(...)`. `paneText(...)` removed.
- `Directory Structure` reorganised around the v2 features:
  `Inbox/`, `Projects/`, `FeatureDetail/`, `Roadmap/`, `Sessions/`,
  `Review/`, `You/`, `Terminal/`.
- `Backend Contract` ▸ `API Surface` rebuilt from `../api/openapi.yaml`
  in seven tables — System, Projects, Features, Tickets, Acceptance
  Criteria, Docs, Decisions, Review, Activity, Agent Sessions,
  Sessions / panes — with a clear note that the spec is authoritative
  if the tables drift.
- `Type Generation` enumerates the additional generated families:
  Tickets, Acceptance Criteria, Docs, Decisions, Activity, Agent
  Sessions, Review (`TicketDiff`, `FileDiff`, `FileChange`,
  `ReviewActionRequest`).
- `Terminal Tab Requirements` renamed to `Terminal Surface
  Requirements` and rephrased so the behaviours stay but the "tab"
  framing is gone. Adds the rule that terminal context (project /
  feature / agent session / pane) must be visible and changeable in
  the slim context bar.
- New `## v2 design reference` section in both files links to the
  Claude Design bundle README and to `docs/feature_plans/00-overview.md`.
- `docs/project_plans/mobile_visual_architecture.md` and
  `mobile_gap_analysis.md` get a one-paragraph "superseded by
  `docs/feature_plans/`" header. Bodies untouched — they remain a
  historical record.
- The first commit on the branch landed the standing
  infra-0003 ticket-move + workhistory bundle (the cleanup pattern
  every branch picks up from its predecessor).

## Decisions

- **Keep `AGENTS.md` and `claude.md` as near-duplicates.** They
  diverge by one or two paragraphs at most; the convention is "edit
  both, never let them drift". A previous attempt to dedupe (symlink /
  generator) introduced more confusion than it removed. Captured in
  the ticket so the same temptation doesn't return.
- **API tables, not free-form prose.** The spec is the source of
  truth and the file already calls that out. Tables make the surface
  scannable and obvious when something is missing — and reviewers
  checking "is this endpoint already in OpenAPI?" can answer in one
  glance instead of reading paragraphs.
- **Old project-plans docs annotated, not deleted.** `mobile_gap_analysis.md`
  and `mobile_visual_architecture.md` are referenced from existing
  workhistory and PR descriptions; deleting them would break links.
  A header note keeps them findable without misleading new readers.
- **`Terminal Surface Requirements`, not `Terminal Drill-down
  Requirements`.** The terminal *is* a surface (its own dark chrome,
  hidden tab bar, slim context bar) regardless of how it was reached.
  "Drill-down" describes the navigation path, not the screen itself.
