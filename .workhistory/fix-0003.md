# fix-0003: Xcode 26.4 build recovery

Ticket: `.tickets/fix-0003-xcode-26-4-build.md`
Retrospective: `docs/retrospectives/2026-05-09-xcode-26.4-recovery.md`

## Summary

Recovered the `remote-coding` app from a wall of compiler errors under Xcode
26.4 / iOS 26.4. Touched 19 source files and 5 test files. Build is green.
Test execution is blocked by an unrelated simulator-launch issue (`SBMainWorkspace`
request denied / `RBSRequest Code=5 Launch failed`) likely caused by two
ENOSPC events that occurred mid-recovery.

## Changes

- Runestone 0.3 → 0.5.2 API migration. `delegate` → `editorDelegate`,
  `font`/`textColor` → `theme`, `lineWrappingEnabled` →
  `isLineWrappingEnabled`, `contentInset` → `textContainerInset`. Removed
  the deleted `text:` binding from call sites.
- OpenAPI codegen audit. Unwrapped `Ticket.estimate` (now non-optional).
  Added `getAgentSession(id:)` to the preview-only `EmptyInboxRepository`
  stub.
- Fixed SwiftUI `@State var body` shadowing in `FeatureDecisionsTab` and
  `ReplySheet`.
- Removed 52 smart-quote characters from a string literal in
  `MockTmuxAgentRepository.swift`.
- Test target catch-up. Replaced renamed calls into `TerminalViewModel`
  (`configure` / `sendEnter` / `sendKey`), added missing `import SwiftUI`
  and `@MainActor` annotations.
- Long-tail Swift fixes — `String(scalars[...])` to a supported initializer,
  missing `Self.` on static refs, `viewModel.diff?.branch.map { ... }`
  accidentally mapping over a `String`'s characters, missing `import
  Runestone` in a file using it.
- Filed `docs/retrospectives/2026-05-09-xcode-26.4-recovery.md` (themed
  root-cause analysis, what went well, what didn't, prioritised action
  items).

## Decisions

- **Single ticket, not three.** The retrospective explicitly raised whether
  this should split into Runestone migration / OpenAPI caller audit /
  cleanup. Chose one ticket because the changes were small in each bucket
  and only meaningful together — the build is the gate, and the build needs
  all three. Splitting would have produced three PRs each of which left the
  tree in a half-broken state. A future *planned* dep bump should still get
  its own ticket per action item #7.
- **Re-home onto `fix-0003` after the fact.** The recovery happened on
  `main` without a branch (workflow violation). Rather than rewrite history,
  carried the working tree onto `fix-0003-xcode-26-4-build` via `branch-new
  --allow-dirty`, then committed cleanly there. `main` was already 2 commits
  behind `origin/main` (phases-6-7 planning merge from PR #44); those
  incoming commits only added files in `.tickets/` and `docs/feature_plans/`
  with no overlap with the recovery diff.
- **Skipped `/ios-gates`, ran build-only.** `/ios-gates` runs `build test`
  and would hang on the wedged simulator. Verified `xcodebuild build`
  succeeds; surfaced the test infra blocker in the PR body and as an open
  question in the retrospective rather than masking it.
- **No safety bypasses.** No `--no-verify`, no `xcrun simctl erase all`,
  no `defaults delete` against simulator daemons without explicit user
  approval. The recovery diff is exclusively code changes.

## Notes

The retrospective's prioritised action items (`/preflight` disk check,
SwiftLint rule for curly quotes and shadowed `body` properties, deletion
of preview-only `Empty*Repository` stubs, iOS CI on PR, codegen-change
caller audit checklist, dedicated branch for forced dep bumps, simulator
auto-recovery, theme-first recovery) are intentionally **not** done in this
ticket — they each warrant their own scoped change against a green build.
File follow-up tickets when picking them up.

The simulator launch failure is the next obvious follow-up. If it persists
after a daemon restart, the action-item #8 retry-once approach in
`/ios-gates` isn't enough; a fuller `simctl shutdown all` + DerivedData
clean cycle may be needed.
